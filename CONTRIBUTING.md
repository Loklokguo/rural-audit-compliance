# 贡献指南

感谢您对 **rural-audit-compliance** 项目的关注！我们欢迎来自社区的贡献。

## 📋 贡献方式

### 1. 报告问题（Issue）

如果您发现问题或有改进建议，请：

1. 检查是否已有相同的 Issue
2. 如果没有，请创建新的 Issue，包含：
   - 问题描述
   - 您的环境信息
   - 复现步骤（如适用）
   - 预期结果 vs 实际结果

### 2. 提交更改（Pull Request）

#### 准备工作

1. Fork 本项目
2. 创建新的分支：`git checkout -b feature/your-feature-name`
3. 进行更改
4. 提交提交：`git commit -m "描述您的更改"`
5. 推送到远程：`git push origin feature/your-feature-name`
6. 提交 Pull Request

#### 提交规范

- **提交消息**：清晰简明地描述更改内容
- **分支命名**：
  - `feature/*` - 新功能
  - `bugfix/*` - 错误修复
  - `docs/*` - 文档更新
- **代码风格**：保持与现有代码一致

### 3. 贡献内容类型

#### ✅ 欢迎的贡献

- 📝 **文档改进** - 修复错别字、完善说明、补充示例
- 📋 **清单优化** - 补充新的审查要点、完善检查项
- 🔍 **指南完善** - 补充风险识别指南、审计方法
- 📜 **政策更新** - 补充新的政策法规参考
- 🐛 **错误修复** - 修复发现的问题
- 💡 **改进建议** - 提出逻辑优化、结构优化建议

#### ❌ 不接受的内容

- 具体地域、单位、人员信息
- 具体金额、账号等敏感数据
- 未脱敏的真实案例
- 违反相关法律法规的内容

## 📖 内容脱敏要求

**所有贡献内容必须进行脱敏处理**：

### 需要脱敏的信息

| 敏感信息 | 脱敏处理 | 示例 |
|--------|--------|------|
| 具体地域 | 用"[地域名称]"替代 | "A省"、"[地级市]" |
| 具体单位 | 用"[单位名称]"替代 | "[乡镇政府]"、"[水利局]" |
| 人员姓名 | 用"[人员名称]"替代 | "[项目负责人]" |
| 具体金额 | 用"[金额]"替代 | 项目投资"[金额]"元 |
| 具体日期 | 保留年份，月日替代 | "2025年[XX月XX日]" |
| 账户信息 | 用"[账户]"替代 | "[交易账户]" |

### 脱敏示例

**❌ 不规范示例**：
```
某县农业局在2025年5月15日，将500万元农业补贴拨付至李某的账户2500010xxxx1234，
引发...争议。
```

**✅ 规范脱敏示例**：
```
[县]农业部门在2025年[X月X日]，将[金额]元农业补贴拨付至[相关账户]，
引发...争议。
```

## 🔄 审核流程

1. **自动检查**：提交 PR 后，系统会进行自动检查（格式、脱敏等）
2. **内容审查**：项目维护者进行内容审查
3. **反馈与修改**：如有需要，我们会提出反馈意见
4. **合并**：通过审查后合并到主分支

## 📋 Pull Request 清单

提交 PR 前，请确保：

- [ ] 我已阅读并同意本贡献指南
- [ ] 我的更改遵循项目的代码风格
- [ ] 我的更改包含适当的文档/注释
- [ ] 所有内容已进行脱敏处理
- [ ] 我没有泄露任何敏感信息
- [ ] 我的 PR 描述清晰明了

## 🎯 优先贡献方向

我们特别欢迎以下方向的贡献：

1. **审计指南扩展** - 补充行业特定的审计指南
2. **风险库完善** - 补充新的风险点和应对措施
3. **清单优化** - 基于实务经验的清单改进
4. **示例完善** - 补充脱敏版的审计案例
5. **国际化** - 文档的英文或其他语言翻译

## ❓ 常见问题

### Q: 我可以贡献真实案例吗？
A: 可以，但必须进行严格的脱敏处理。所有具体信息都应替换为通用术语。

### Q: 如果我的 PR 被拒绝怎么办？
A: 我们会提供具体的反馈意见。您可以根据反馈进行修改后重新提交。

### Q: 贡献内容是否会被使用？
A: 所有被接受的贡献都会在项目中使用，并在适当的位置标注贡献者。
